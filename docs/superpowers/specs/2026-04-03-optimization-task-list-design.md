# 优化任务列表页面 UI 优化设计

## 1. 背景

当前“优化任务列表”页已经具备基础能力：

- 查看历史优化任务
- 按类型筛选 SQL / MyBatis 任务
- 进入创建 SQL 优化任务
- 进入创建 MyBatis XML 优化任务
- 查看任务详情与优化结果

但当前页面仍然偏“功能集合页”，存在几个明显问题：

- 顶部两个创建入口只是大按钮，虽然显眼，但信息层次单薄，难以承担“本页主入口”的角色
- 页面没有明确区分“创建新任务”和“查看历史任务”两类意图，视觉节奏较平
- 任务列表具备信息，但缺少状态概览，页面无法在首屏快速传达当前任务运行情况
- 表格卡片、筛选区、入口区之间风格还没有和近期调整过的登录页、权限管理页统一
- 当列表无数据时，页面缺少更主动的引导，首页容易显得空

本次优化只针对“优化任务列表”这一页，目标是在不改后端接口和核心交互流程的前提下，让它同时承担两类职责：

- 作为创建新任务的清晰入口
- 作为查看任务状态与历史结果的轻量工作台

## 2. 目标

### 2.1 功能目标

- 让“新建任务入口”成为首屏最明确的主要动作
- 保留现有任务列表、筛选、刷新、分页、详情跳转能力
- 在首屏补充轻量状态概览，提升任务监控感
- 在无数据场景下提供明确的下一步引导

### 2.2 体验目标

- 风格统一到当前系统的浅色、干净、专业、非装饰化后台方向
- 页面层次更清楚，让用户进入页面后立刻知道“先创建任务”或“继续查看结果”
- 保持后台系统效率感，不做花哨报告页，也不做重监控大屏

### 2.3 非目标

- 不改任务创建流程
- 不改详情页结构
- 不新增后端统计接口
- 不新增复杂筛选维度或批量操作

## 3. 用户已确认的方向

本次设计经过收敛，已确认如下：

- 只优化 `优化任务列表` 单页
- 页面需要同时兼顾 `任务监控总览` 和 `进入创建/查看结果的工作台入口`
- 首屏优先级：`新建任务入口优先`
- 方案方向：`A 入口优先型`

## 4. 总体方案

页面采用“三段式布局”：

- 顶部：页面标题与说明
- 中部：两个高优先级的新建任务入口卡片
- 下部：轻量状态概览 + 任务列表主卡片

整体体验从“按钮 + 表格”升级为“入口优先的任务工作台”。用户在进入页面后，能够迅速回答三个问题：

- 我要从哪里创建新任务
- 当前任务大致处于什么状态
- 我如何继续查看已有任务结果

## 5. 页面结构设计

## 5.1 页面头部

保留页面标题区，但增强副标题的任务导向：

- 标题：`优化任务列表`
- 副标题：`创建新的优化任务，并跟踪历史结果与状态`

头部不增加面包屑，不增加复杂说明段落，保持短句和清晰开场。

## 5.2 新建任务入口区

这是本页首屏最重要的区域，位于标题下方，使用两个并列的大卡片替代当前纯按钮：

- `优化单个 SQL 查询`
- `优化 MyBatis XML 文件`

每个入口卡片包含：

- 标题
- 一句简短说明
- 明确的主操作提示

交互规则：

- 整个卡片可点击，不仅仅是内部按钮区域
- hover 仅做边框、阴影和背景变化，不做明显位移或夸张动画
- SQL 入口保持第一主入口视觉层级
- MyBatis 入口保持同级，但在文案上区分适用场景

设计目标：

- 用户进入页面时，第一眼就知道“这里可以开始一个新的优化任务”
- 入口本身就能传达使用差异，而不是只靠按钮文字

## 5.3 状态概览区

入口区下方增加一排轻量状态概览卡片，用来补齐页面的监控属性。

建议展示四项：

- 全部任务
- 优化中
- 已完成
- 失败

实现原则：

- 统计数据直接基于当前 `store.list` 在前端计算
- 只做信息展示，不引入新的后端接口
- 默认不做点击筛选联动，避免增加交互复杂度

视觉原则：

- 比入口卡片更轻
- 更像摘要信息，而不是 KPI 大屏
- 用统一但克制的状态色表达不同任务状态

## 5.4 任务列表主卡片

页面下半部分保留任务表格，但升级为结构更完整的主内容区。

主卡片分为三层：

- 顶部：标题与任务总数
- 工具区：类型筛选 + 刷新按钮
- 内容区：任务表格 + 分页

表格保留现有字段：

- 类型
- 对象
- 状态
- 数据库 IP
- 库名
- 创建时间
- 操作

但视觉重点会重新分配：

- `对象` 作为主信息展示，强化可点击性
- `状态` 作为第二重点，统一更克制的标签风格
- `查看` 操作弱化为辅助入口，避免与对象链接重复抢焦点

## 5.5 空状态

当当前列表为空时，页面不能只显示空表格。

推荐空状态文案：

- `还没有优化任务，先从上方创建一个新任务`

要求：

- 空状态放在表格内容区内部
- 文案明确指向上方入口区
- 让首页在零数据时仍然有明确下一步

## 6. 视觉风格

## 6.1 视觉方向

延续当前系统已建立的浅色企业后台风格：

- 白底卡片
- 浅蓝边框
- 轻阴影
- 非装饰化背景
- 纯色操作按钮

本页不使用：

- 大面积渐变背景
- 高饱和跳色大块
- 炫技式图表视觉
- 重度报告页排版

## 6.2 色彩建议

- 主色：`#0369A1`
- 辅助浅蓝：`#E8F3FB`
- 页面背景：沿用系统浅灰蓝底
- 文本主色：`#0F2A3D`
- 边框：`#D9E6F2`

状态建议：

- 优化中：浅琥珀/蓝灰，避免过亮黄
- 已完成：浅青绿，但保持克制
- 失败：浅红背景 + 深红文字，避免抢主视觉

## 6.3 排版建议

- 页面标题和副标题使用清晰的上下层级
- 新建任务入口卡片内部使用“标题 + 描述 + 操作提示”三层结构
- 状态概览卡片用“数字 + 标签”结构，不堆文字
- 任务表格保持后台信息密度，但加大关键列的可读性和留白

## 7. 交互细节

## 7.1 新建入口交互

- 整卡可点击
- hover 时显示边框强化与轻阴影反馈
- 保持稳定，不使用缩放抖动效果

## 7.2 列表交互

- `对象` 链接保持跳转详情逻辑不变
- `查看` 按钮保留，但作为次级入口
- 刷新按钮保留原逻辑
- 类型筛选保留原逻辑
- 分页保留原逻辑

## 7.3 状态反馈

- 列表加载时维持当前 loading 行为
- 状态概览不单独加 loading 骨架，直接复用列表数据生命周期
- 无数据时显示更明确空状态，不使用无意义的默认空表格文案

## 8. 响应式行为

桌面端：

- 两个入口卡片并列展示
- 状态概览 4 列并排
- 列表卡片完整展开

平板端：

- 入口卡片可保持双列或转为上下布局，视空间而定
- 状态概览可变为 2x2

移动端：

- 两个入口卡片上下堆叠
- 状态概览 2x2
- 工具区改为纵向排列
- 表格允许横向滚动，不强制压缩字段造成难读

## 9. 可访问性与一致性

页面至少满足以下要求：

- 入口卡片和可点击对象有明显 hover / focus 状态
- 主按钮继续使用纯色，不回退到渐变按钮
- 选中或状态表达不只依赖颜色，仍需有文字标签
- 所有交互风格与当前登录页、权限管理页保持同一套企业后台语言

## 10. 实现边界

本次实现只改前端页面层：

- 主要修改文件：`frontend/src/views/OptimizationTaskList.vue`

允许的实现内容：

- 模板结构调整
- 页面级计算属性
- 样式重构
- 页面级测试

不做的内容：

- 不改 store 契约
- 不改后端 API
- 不新增统计接口
- 不改详情页或创建页

## 11. 测试与验证

至少完成以下验证：

- 两个新建任务入口都能正常跳转
- 类型筛选、刷新、分页逻辑不回归
- 状态概览能正确反映当前列表数据
- 空状态文案正确出现
- 前端测试和构建通过
