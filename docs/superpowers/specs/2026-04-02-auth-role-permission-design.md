# 用户登录、角色与权限功能设计

## 1. 背景

当前系统已经具备以下业务模块：

- SQL优化建议
- 慢SQL列表
- 连接管理
- 归档任务
- 执行日志

系统目前没有统一的登录、角色、页面权限和数据权限体系。所有页面和接口默认可访问，无法满足按用户、角色和连接范围控制访问能力的要求。

本次设计目标是在尽量复用现有系统结构的前提下，新增一套最小可用、可持续扩展的账号与权限体系，并为后续企业单点登录接入预留扩展能力。

## 2. 目标

### 2.1 功能目标

- 支持用户通过工号和密码登录系统
- 角色固定为两种：管理员、普通用户
- 管理员可访问全部页面和全部数据
- 普通用户仅可访问指定页面：SQL优化建议、慢SQL列表
- 普通用户只能使用管理员创建并授权的连接
- 普通用户在 SQL优化建议 模块中仅可查看自己创建的数据
- 普通用户在 慢SQL列表 模块中不按创建人过滤，而是按连接授权范围查看数据
- 新增用户管理、角色管理、权限管理页面

### 2.2 非功能目标

- 不推翻现有连接管理功能，直接复用已有连接表和页面
- 认证与权限校验以后端为最终边界，前端仅做体验控制
- 后续可平滑扩展企业 SSO，不要求重写现有认证体系
- 控制设计复杂度，避免引入超出当前需求的通用权限平台

## 3. 范围

### 3.1 本次纳入范围

- 登录与登出
- 当前用户信息获取
- 用户管理
- 角色说明管理
- 用户连接授权管理
- 前端菜单与路由控制
- 后端接口鉴权
- SQL优化建议模块的数据权限接入
- 慢SQL列表模块的数据权限接入

### 3.2 本次不纳入范围

- 企业 SSO 真实对接
- 自助注册
- 自助找回密码
- 自定义角色
- 自定义页面权限配置
- 细粒度字段级权限
- 审批流

## 4. 核心约束

- 登录方式先按“工号 + 密码”设计，后续预留 SSO 扩展
- 用户由管理员手工创建
- 用户表至少包含：姓名、工号、部门、角色
- 角色固定仅两种：`admin`、`user`
- 普通用户共用一套固定页面权限
- 普通用户仅可访问：
  - SQL优化建议
  - 慢SQL列表
- 普通用户不允许进入连接管理，也不允许维护连接
- 连接只能由管理员创建和授权
- SQL优化建议模块按“自己创建 + 已授权连接”双重约束过滤
- 慢SQL列表模块不按创建人过滤，只按慢SQL数据中的“主机”字段与连接管理中的“管理IP”字段进行关联过滤

## 5. 总体方案

采用固定角色 + 固定页面权限 + 用户连接授权 + 业务规则过滤的最小可用方案。

### 5.1 页面权限

- 管理员：可访问全部页面
- 普通用户：仅可访问
  - SQL优化建议
  - 慢SQL列表

### 5.2 数据权限

- SQL优化建议
  - 管理员：可查看全部数据
  - 普通用户：仅可查看自己创建、且连接在授权范围内的数据

- 慢SQL列表
  - 管理员：可查看全部数据
  - 普通用户：仅可查看与自己已授权连接匹配的慢SQL数据
  - 匹配规则：`慢SQL主机字段 = 连接管理.manage_host`

### 5.3 认证原则

- 前端负责登录页、菜单显示和路由守卫
- 后端负责登录态、接口鉴权和最终数据过滤
- 任何页面隐藏或菜单隐藏都不能替代后端权限校验

## 6. 数据模型设计

### 6.1 复用现有表：`db_connection`

继续复用现有连接表，不新增连接主表，不重做连接管理页面。

职责不变：

- 管理员维护连接
- 普通用户不可直接操作连接
- 业务模块通过连接 ID 与用户授权关系决定访问范围

### 6.2 新增用户表：`sys_user`

建议字段：

- `id`：主键
- `employee_no`：工号，唯一
- `password_hash`：密码哈希
- `real_name`：姓名
- `department`：部门
- `role_code`：角色，`admin` / `user`
- `status`：状态，`enabled` / `disabled`
- `last_login_at`：最后登录时间
- `last_login_ip`：最后登录 IP
- `created_by`
- `updated_by`
- `created_at`
- `updated_at`
- `deleted_at`：逻辑删除时间，可选但建议保留

设计说明：

- 登录账号使用 `employee_no`
- 密码只存哈希，不存明文
- 角色不单独拆表，避免对固定角色场景过度设计
- 为未来 SSO 预留扩展字段位，例如 `sso_subject`、`sso_source`

### 6.3 新增用户连接授权表：`sys_user_connection_permission`

建议字段：

- `id`
- `user_id`
- `connection_id`
- `status`：`enabled` / `disabled`
- `granted_by`
- `granted_at`
- `remark`
- `created_at`
- `updated_at`

约束建议：

- `user_id + connection_id` 唯一
- 管理员默认拥有全部连接权限，不依赖该表
- 普通用户的数据访问范围以该表为准

### 6.4 新增登录日志表：`sys_login_log`

建议字段：

- `id`
- `user_id`
- `employee_no`
- `login_result`
- `failure_reason`
- `login_ip`
- `user_agent`
- `created_at`

用途：

- 审计登录行为
- 排查异常登录和密码问题
- 为后续密码错误次数限制预留统计基础

### 6.5 补充业务表字段

重点补充 `optimization_task`：

- `creator_user_id`
- `creator_employee_no`
- `connection_id`
- `database_name`

用途：

- 支持按创建人过滤
- 支持按连接授权过滤
- 支持列表页直接展示库名

慢SQL模块不要求按创建人过滤，因此不强依赖创建人字段。

## 7. 页面与交互方案

### 7.1 登录页

字段：

- 工号
- 密码
- 登录按钮

规则：

- 工号和密码均为必填
- 被禁用用户不可登录
- 登录失败统一提示“工号或密码错误”，不区分工号不存在还是密码错误

登录成功后的跳转：

- 管理员：进入系统默认首页
- 普通用户：进入 SQL优化建议 页面

### 7.2 用户管理页面

仅管理员可访问。

列表字段：

- 姓名
- 工号
- 部门
- 角色
- 状态
- 最近登录时间
- 创建时间
- 操作

支持操作：

- 新增用户
- 编辑用户
- 启用用户
- 禁用用户
- 重置密码
- 删除用户

表单字段：

- 姓名
- 工号
- 部门
- 角色
- 初始密码
- 状态

约束：

- 工号唯一
- 删除建议采用逻辑删除

### 7.3 角色管理页面

仅管理员可访问。

由于角色固定为两种，该页面定义为系统内置角色说明页，而不是角色配置页。

展示内容：

- 管理员角色说明
  - 可访问全部页面
  - 可查看全部连接及全部数据
  - 可管理用户与授权
- 普通用户角色说明
  - 仅可访问 SQL优化建议、慢SQL列表
  - SQL优化建议仅可查看自己创建且连接已授权的数据
  - 慢SQL列表仅按授权连接查看

不提供：

- 新增角色
- 删除角色
- 编辑角色编码

### 7.4 权限管理页面

仅管理员可访问。

页面定位：用户连接授权管理。

推荐结构：

- 用户信息区
  - 姓名
  - 工号
  - 部门
  - 角色
  - 状态

- 连接授权区
  - 已授权连接列表
  - 未授权连接列表
  - 支持按连接名称、IP 搜索
  - 支持授权、取消授权

原则：

- 管理员默认拥有全部连接权限，不在此页配置
- 普通用户的连接使用权限完全由此页维护

### 7.5 菜单控制

管理员显示全部菜单。

普通用户仅显示：

- SQL优化建议
- 慢SQL列表

普通用户不可见：

- 连接管理
- 归档任务
- 执行日志
- 用户管理
- 角色管理
- 权限管理

## 8. 鉴权与访问控制设计

### 8.1 登录态方案

建议采用 Cookie Session 模式。

原因：

- 当前前后端部署模式适合同域 Cookie
- 不需要把 token 存在浏览器本地存储
- 安全边界更清晰
- 后续接入 SSO 更容易扩展

Session 中仅保存最小必要信息：

- `user_id`
- `employee_no`
- `role_code`

每次接口请求由后端重新读取用户状态，确保：

- 用户被禁用后可立即失效
- 用户角色变化后可立即生效

### 8.2 后端鉴权能力拆分

建议统一抽出以下能力：

- `get_current_user()`
- `login_required`
- `admin_required`
- `connection_scope_filter`

目的：

- 避免在每个接口中复制权限判断
- 便于各业务模块复用统一规则

### 8.3 前端控制

前端新增 `auth store`，统一管理：

- 当前用户信息
- 是否登录
- 当前角色
- 可见菜单
- 登录、登出动作

前端新增路由守卫：

- 未登录访问业务页，跳转登录页
- 普通用户访问管理员页面，跳转 SQL优化建议 页

### 8.4 后端最终权限边界

所有业务接口必须以后端为最终边界。

后端必须验证：

- 是否已登录
- 当前角色是否允许访问当前功能
- 当前用户是否拥有目标连接权限
- 当前资源是否满足创建人归属规则

## 9. 业务模块接入方案

### 9.1 SQL优化建议

普通用户：

- 创建任务时只能选择已授权连接
- 查询列表时只返回自己创建的数据
- 查看详情时需同时满足：
  - 任务由自己创建
  - 任务连接在授权范围内

管理员：

- 可查看全部任务
- 可选择全部连接

### 9.2 慢SQL列表

普通用户：

- 不按创建人过滤
- 仅可查看其已授权连接对应的慢SQL数据

过滤规则：

1. 获取该用户已授权连接
2. 取这些连接的 `manage_host`
3. 用 `manage_host` 匹配慢SQL数据中的“主机”字段
4. 仅返回匹配到的慢SQL记录

管理员：

- 可查看全部慢SQL数据

### 9.3 其他现有业务模块

以下模块管理员可访问，普通用户不可访问：

- 连接管理
- 归档任务
- 执行日志

后端接口必须显式增加管理员校验，不能只依赖前端隐藏菜单。

## 10. 接口设计建议

### 10.1 认证接口

- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`

`/api/auth/login` 返回内容建议包含：

- 当前用户基础信息
- 角色信息
- 可见菜单

### 10.2 用户管理接口

- `GET /api/admin/users`
- `POST /api/admin/users`
- `PUT /api/admin/users/:id`
- `PUT /api/admin/users/:id/status`
- `PUT /api/admin/users/:id/reset-password`
- `DELETE /api/admin/users/:id`

### 10.3 角色管理接口

- `GET /api/admin/roles`

用途：

- 返回系统内置两种角色及其权限说明

### 10.4 权限管理接口

推荐按“用户整体授权”方式设计：

- `GET /api/admin/user-connection-permissions?user_id=:id`
- `PUT /api/admin/user-connection-permissions/:user_id`

`PUT` 接口一次性提交该用户的全部授权连接列表，便于前端维护和审计。

## 11. 异常处理规则

- `401`：未登录或登录态失效
- `403`：无页面权限、无连接权限或无数据权限
- `404`：资源不存在，或资源不在当前用户可见范围内
- `400`：参数错误
- `409`：数据冲突，例如工号重复、重复授权

统一要求：

- 登录失败不暴露具体原因
- 权限不足必须返回明确错误码
- 前端根据状态码进行统一跳转或提示

## 12. 实施顺序

### 阶段 1：数据层改造

- 新增 `sys_user`
- 新增 `sys_user_connection_permission`
- 新增 `sys_login_log`
- 补充 `optimization_task` 权限相关字段

### 阶段 2：认证层

- 登录
- 登出
- 当前用户信息
- Session 管理
- 登录拦截器与管理员拦截器

### 阶段 3：前端登录与菜单

- 登录页
- `auth store`
- 路由守卫
- 动态菜单显示

### 阶段 4：管理员功能

- 用户管理页
- 角色管理页
- 权限管理页

### 阶段 5：业务权限接入

- SQL优化建议模块
- 慢SQL列表模块
- 管理员专属页面统一加保护

### 阶段 6：测试与验收

- 登录测试
- 页面权限测试
- 数据权限测试
- 禁用用户测试
- 连接授权测试
- 越权访问测试

## 13. 验收标准

- 管理员登录后可访问全部页面
- 普通用户登录后仅看到 SQL优化建议、慢SQL列表
- 普通用户不能进入连接管理、归档任务、执行日志、用户管理、角色管理、权限管理
- 管理员可新增、编辑、禁用、启用、删除用户
- 管理员可为普通用户分配连接权限
- 普通用户创建 SQL优化任务时只能选择已授权连接
- 普通用户在 SQL优化建议 中只能看到自己创建的数据
- 普通用户在 慢SQL列表 中只能看到已授权连接对应的数据
- 普通用户在 慢SQL列表 中的数据过滤只依据：
  - 慢SQL主机字段
  - 连接管理中的管理IP字段
- 用户被禁用后立即无法继续访问系统
- 未登录、越权、无连接权限三类场景均可返回明确错误

## 14. 扩展预留

### 14.1 SSO 扩展

后续接入企业 SSO 时，可保留现有用户表，新增 SSO 标识字段，并将登录入口从“工号 + 密码”扩展为：

- 本地账号密码登录
- 企业 SSO 登录

两者共享同一套角色和数据权限体系。

### 14.2 页面权限扩展

当前普通用户页面权限是固定写死的。若后续需要做到“不同普通用户可访问不同页面”，可在现有结构上继续扩展：

- 新增菜单权限表
- 为角色或用户绑定菜单权限

不影响当前连接授权和数据权限模型。

## 15. 结论

本方案采用最小可用、可扩展、与现有系统兼容的设计路线：

- 角色固定两种，不做过度抽象
- 页面权限固定，满足当前明确业务边界
- 连接权限独立建模，满足后续业务扩展
- SQL优化建议和慢SQL列表分别按各自业务规则过滤
- 后端作为最终权限边界，前端负责菜单与体验控制

该方案能够满足当前需求，并为后续 SSO 接入和更细粒度权限扩展保留清晰演进路径。
